Clockchain developer tools

A CLI and an MCP server over the existing D4 node API, so developers and AI agents can use the time oracle without writing raw HTTP calls.

Yang Tang · June 2026 · companion to specs/implementation-plan.md

Where things stand today

  • The time oracle works. The public gateway answers three time endpoints with an API key.
  • There is no CLI, no SDK, no MCP server. The only way in is curl.
  • The assessment sets a goal of "time-to-first-log under x minutes" and leaves x blank. The job here is to make that number real and small.
# the entire current developer experience
curl -H "x-api-key: ..." https://node.clockchain.network/api/time/time

What this builds

Three npm packages over one shared client. Nothing here changes the blockchain protocol.

@clockchain/coreHTTP client, types, DID helpers, evidence assembly. Everything else depends on it.
@clockchain/cliFor developers at a terminal. npx @clockchain/cli.
@clockchain/mcp-serverFor AI agents. The integration surface for the Agent-SDK (Product B).

All TypeScript: the MCP SDK and LangChain.js live in that ecosystem, and one language keeps the team's surface area small.

The two products these serve

Product A - Agent identity

On-chain DIDs for agents: mint, verify, revoke. A "birth certificate" an agent can point to.

Product B - Agent-SDK

The path for LangChain / AutoGPT / CrewAI agents to log actions and read consensus time. The MCP server is how they connect.

Both are built as a convention layer on the existing /log and /searchAsset APIs. See the status slide for what that convention can and cannot do yet.

What the gateway actually exposes

Probed with a live key 2026-06-02; write path verified on a funded account 2026-06-03. This is the ground the plan stands on.

EndpointResult
/api/time/time, /timestamp, /blockconfirmed wrapped in {success,data,meta}
/searchAssetverified returns the logged entry; scoped to caller's clientId
/getValidationBlock/{h}confirmed no envelope; testnet votes are all zero
/log, /ledger/{id}verified end-to-end wrote, anchored at block 2822102 in ~0.6s, read back. walletId = account email.
/schedule404 not on the gateway - smart-contract triggers are blocked

Response envelopes are inconsistent across endpoints. The client normalizes per-endpoint rather than assuming one wrapper.

Two things to settle with the D4 team

  1. Cross-client lookup. searchAsset only sees your own records. Two agents on different clients can't verify each other's identity, which is the point of a handshake. Needs a public /did/{id} resolver.
  2. Triggers. /schedule isn't proxied to the gateway. The whole time-conditional-contract feature waits on that.

Resolved already: walletId = the account email works for writes, and logging is verified end-to-end. Neither remaining item blocks the read-and-log core - they block the parts we'd otherwise oversell.

The flow an agent follows

  1. Mint - get a DID, recorded as a log entry
  2. Log - every meaningful action, hashed and timestamped against that DID
  3. Prove - assemble the consensus + ledger record for an entry
  4. Verify - check a DID is present and not revoked (same client, for now)
  5. Revoke - retire the identity

The MCP onboard_agent prompt walks a new agent through steps 1-3 on first connect.

CLI

$ clockchain identity mint --name investor-update-agent \
    --capabilities "draft,send-email,read-metrics"
  DID: did:clockchain:agent:a8f3c2e1...   block: 2734500

$ clockchain log file ./update.pdf --did did:clockchain:agent:a8f3c2e1... \
    --asset-ref update-2026-06-02 --wait
  ledger: b9c4d3f2...   block: 2734510   status: confirmed

$ clockchain prove b9c4d3f2...
  consensus + ledger layers populated; layers 1-3 marked unverifiable
  (testnet: single node, votes show 0 - expected)

P0 set: auth, identity mint/verify, log/search, prove, time. That is the five-minute path, start to first logged action.

On "court-admissible"

The audit document lays out a five-layer evidence chain. Worth being precise about what we can build today:

LayerBuildable now
1 - VRF validator electionno - no endpoint returns the election proof
2 - per-validator time readingsno - not exposed
3 - independent computationdescribed only
4 - validator consensus (votes, trust)yes - getValidationBlock
5 - ledger recordyes - ledger + block

So the honest deliverable is a two-layer proof. The legal standards (eIDAS, RFC 3161) are the destination once a real validator set and the missing endpoints exist - not what a single-node testnet entry demonstrates. Say that plainly to customers.

Why subnets matter later

Mainnet does ~50K txns/day. One enterprise customer generates millions of log events a day. You can't put raw logs on a shared chain.

customer logs  →  dedicated subnet (3-30 nodes)  →  one hash per cycle  →  mainnet
  • Logs stay private in the subnet; only a state-root fingerprint is anchored.
  • Mainnet load scales with customer count, not log volume.

Backend status: spawn/assign/dissolve handlers done; customer-assignment API in progress; subnet-internal time consensus planned Q3 2026. The tools gain --subnet routing when that lands.

How it's tested

  • Unit - mocked client, runs on every commit, no network (~2s)
  • Protocol - spawn the MCP server, drive it with a real client over stdio
  • Smoke / integration - against the live testnet, run before a release
  • Contract - snapshot the API shapes, diff weekly, catch D4 changes early

A release cycle costs ~50 testnet requests. The free plan's 1,000 covers roughly twenty cycles, so quota isn't a constraint at this stage.

What "done" looks like

  • A developer goes from npx to a logged action in under five minutes.
  • An agent in Claude Code mints a DID and logs its first action from one config block.
  • The package is listed where agent developers look (mcp.run, glama.ai).
  • Five outside testers have run it - the activation number the assessment asked for.

Honest framing throughout: we ship the read-and-log core that works today, and we name the identity-resolution and trigger pieces as backend dependencies rather than implying they're finished.
